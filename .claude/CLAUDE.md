# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

---

## What This Repository Is

**batuta-dots** is the Batuta ecosystem hub. Its job is to produce and distribute configuration files — `CLAUDE.md`, skills, agents, commands, hooks, and settings — to client projects. You are working on the hub itself, not a project that uses it.

`BatutaClaude/CLAUDE.md` is the tracked canonical source and the artifact that gets deployed to client projects. Root `CLAUDE.md` is a **gitignored** generated mirror regenerated by `setup.sh --claude` for developer convenience (so the agent working inside batuta-dots reads the current content from the repo root). Edit `BatutaClaude/CLAUDE.md`, then run `bash infra/setup.sh --claude` to refresh the root mirror. Every downstream project that runs `/batuta-update` pulls from `BatutaClaude/CLAUDE.md`.

## Development Commands

```bash
# Syntax-check all scripts (non-destructive, safe to run anytime)
bash -n infra/setup.sh
bash -n infra/sync.sh
bash -n infra/hooks/session-start.sh
bash -n infra/hooks/session-save.sh

# Run the full test suite
bash infra/setup_test.sh

# Verify current installation against what's in the repo
bash infra/setup.sh --verify

# Push changes to the global Claude installation (your own machine)
bash infra/setup.sh --all          # CLAUDE.md + skills + hooks
bash infra/setup.sh --sync         # Skills + agents (fastest)
bash infra/setup.sh --hooks        # Hooks + settings.json only
bash infra/setup.sh --claude       # CLAUDE.md only

# Test a specific project setup (non-destructive if <path> is a temp dir)
bash infra/setup.sh --project <path>
```

## Key Invariants

These are the rules for modifying the hub. Breaking any of them will cause downstream project failures.

### BatutaClaude/CLAUDE.md is canonical; root CLAUDE.md is a generated mirror
`BatutaClaude/CLAUDE.md` is tracked in git and consumed by `setup.sh --project` and `/batuta-update` to deploy to client projects. Root `CLAUDE.md` is in `.gitignore` — it exists only as a developer-convenience mirror regenerated by `setup.sh --claude` (copies `BatutaClaude/CLAUDE.md` → root). **Always edit `BatutaClaude/CLAUDE.md`**, then run `bash infra/setup.sh --claude` to refresh the root mirror. A guardrail test (`test_claude_md_is_direct_copy` in `setup_test.sh`) enforces that the two files are byte-identical to catch drift.

### SKILL.md frontmatter is mandatory
Every skill in `BatutaClaude/skills/{name}/SKILL.md` requires a YAML frontmatter block. The `session-start.sh` hook parses it with awk — missing or malformed frontmatter silently drops the skill from the inventory.

Required frontmatter fields:
```yaml
---
name: skill-name
description: >
  Use when ... Trigger: "keyword1", "keyword2".
metadata:
  scope: [pipeline]          # pipeline | infra | observability
  platforms: [claude, antigravity]
allowed-tools: Read Edit Write Glob Grep Bash
---
```

The `description` field must follow the `"Use when... Trigger: ..."` convention — the session-start hook and Batuta Bootstrap depend on this pattern for skill activation.

### VERSION file controls update detection
`BatutaClaude/VERSION` is a plain-text file containing only the semver string (e.g., `14.0.0`). `session-start.sh` compares this against `.batuta/ecosystem.json` in client projects to detect drift. Increment it when making breaking changes to the hub API (CLAUDE.md structure, skill contracts, agent interfaces).

### Agents go in `BatutaClaude/agents/`
Agent files are plain Markdown. The three scope agents (pipeline, infra, observability) handle SDD routing. The three domain agents (backend, data, quality) are invoked by the main agent via the Task tool. Do not create a fourth domain agent unless the new domain has its own conventions, 3+ dedicated skills, and clear scope boundaries.

### Commands exist in two places for cross-platform support
Slash commands that support both Claude and Antigravity must exist in:
- `BatutaClaude/commands/{name}.md` (Claude Code)
- `BatutaAntigravity/workflows/{name}.md` (Antigravity/Gemini)

## Testing Changes

Before opening a PR for any hub change:

1. **Syntax check** — `bash -n infra/setup.sh && bash -n infra/hooks/session-start.sh`
2. **Full test suite** — `bash infra/setup_test.sh`
3. **Smoke test** — `bash infra/setup.sh --all` on your own machine, then open a test project and verify the skill inventory appears at session start

The test suite (`setup_test.sh`) validates:
- SKILL.md frontmatter structure across all 38 skills
- Hook file paths in settings.json
- VERSION file existence
- Required CLAUDE.md sections (Execution Gate, Session Continuity, SDD Pipeline, etc.)
- Team templates count
